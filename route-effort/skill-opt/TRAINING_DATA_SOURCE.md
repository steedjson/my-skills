### 数据创建方法

这些训练数据是**人工构造的真实任务描述**，基于以下来源：

#### 1. 常见开发任务

从日常软件开发中提取的典型任务，覆盖 5 个 effort 级别：

**Low 级别样本（机械操作）**:
```json
{
  "id": "train-0",
  "query": "格式化 utils.py 文件",
  "expected_effort": "low"
}
{
  "id": "train-1",
  "query": "重命名变量 userName 为 username",
  "expected_effort": "low"
}
```

**Medium 级别样本（单文件任务）**:
```json
{
  "id": "train-5",
  "query": "修复单个文件中的 bug",
  "expected_effort": "medium"
}
{
  "id": "train-7",
  "query": "实现一个简单的工具函数",
  "expected_effort": "medium"
}
```

**High 级别样本（多文件任务）**:
```json
{
  "query": "实现用户认证系统",
  "expected_effort": "high"
}
{
  "query": "重构数据库访问层",
  "expected_effort": "high"
}
```

**XHigh 级别样本（跨模块任务）**:
```json
{
  "query": "重构支付流程",
  "expected_effort": "xhigh"
}
{
  "query": "分析系统性能瓶颈",
  "expected_effort": "xhigh"
}
```

**Max 级别样本（高风险任务）**:
```json
{
  "query": "审计加密密钥管理",
  "expected_effort": "max"
}
{
  "query": "修复并发竞态条件",
  "expected_effort": "max"
}
```

#### 2. 真实项目经验

样本反映了真实软件开发中的任务类型：
- 代码格式化和重命名（低复杂度）
- Bug 修复和功能实现（中等复杂度）
- 架构重构和系统设计（高复杂度）
- 安全审计和并发问题（最高复杂度）

#### 3. 覆盖边界案例

数据集特意包含了易混淆的边界案例：
- "设计缓存失效策略" - 可能是 high 或 xhigh
- "实现用户登录功能" - 可能是 medium 或 high
- "重构支付流程" - 可能是 xhigh 或 max

这些边界案例用于测试模型的判断能力。

## 数据格式

每个样本包含 3 个字段：

```json
{
  "id": "train-0",           // 唯一标识符
  "query": "任务描述",       // 中文任务描述
  "expected_effort": "low"   // 预期的 effort 级别
}
```

### Effort 级别定义

| 级别 | 定义 | 示例 |
|------|------|------|
| low | 机械性操作，无需推理 | 格式化、重命名、删除 |
| medium | 单文件任务，需少量推理 | Bug 修复、单函数实现 |
| high | 多文件任务，需方案选择 | 功能开发、架构分析 |
| xhigh | 跨模块任务，影响面大 | 系统重构、根因分析 |
| max | 高风险，错误代价极高 | 安全审计、并发 bug |

## 数据质量

### 优点

1. **覆盖全面**: 包含所有 5 个 effort 级别
2. **真实性**: 基于实际开发任务
3. **中文描述**: 符合中文开发者的使用习惯
4. **结构清晰**: JSON 格式，易于解析

### 不足

1. **数量有限**: 
   - 训练集 40 样本（建议 100+）
   - 验证集 5 样本（建议 10+）
   - 测试集 5 样本（建议 10+）

2. **人工标注**: 
   - 可能存在主观判断差异
   - 缺少多人标注的一致性验证
   - 没有标注者间一致性（Inter-rater Agreement）分数

3. **边界模糊**: 
   - 某些任务的级别判断有争议
   - 缺少明确的判断标准文档

4. **缺乏多样性**:
   - 样本可能过于理想化
   - 缺少"奇怪"或"意外"的真实查询
   - 没有包含拼写错误、不完整描述等真实情况

## 数据扩展建议

### 短期扩展（目标 100 样本）

从以下来源收集真实数据：

1. **Git Commit 消息**
```bash
# 提取最近 100 个 commit 的消息
git log --oneline -100 > commits.txt
# 手动标注 effort 级别
```

2. **Issue/PR 标题**
```bash
# 使用 gh/glab 提取
gh issue list --limit 100 --json title
# 手动标注
```

3. **团队聊天记录**
- 从 Slack/微信/钉钉中提取任务描述
- 匿名化后标注

### 中期扩展（目标 500+ 样本）

1. **众包标注**
   - 邀请多位开发者独立标注
   - 计算标注者间一致性
   - 使用投票或专家复核解决分歧

2. **在线收集**
   - 在生产环境中收集真实查询
   - 记录模型预测和用户反馈
   - 构建"难样本"数据集

3. **数据增强**
   - 同义词替换（"修复" → "解决"）
   - 添加上下文（"在支付模块中修复 bug"）
   - 生成边界案例

### 长期维护

1. **持续更新**
   - 每月添加新样本
   - 定期重新评估旧样本
   - 根据生产反馈调整标签

2. **质量控制**
   - 建立数据审核流程
   - 定期计算数据集统计指标
   - 监控数据分布变化

3. **版本管理**
   - 使用语义化版本号（v1.0.0）
   - 记录每次数据更新的 changelog
   - 保留历史版本用于对比实验

### 数据标注指南

为新任务标注 effort 级别时，参考这个决策树：

```
1. 是否机械性操作，无需推理？
   → 是：low
   → 否：继续

2. 是否单文件任务，边界清晰？
   → 是：medium
   → 否：继续

3. 是否多文件任务，有歧义或需方案选择？
   → 是：high
   → 否：继续

4. 是否跨模块，影响面大？
   → 是：xhigh
   → 否：继续

5. 是否安全/并发/正确性至关重要？
   → 是：max
   → 否：向上路由一级（保守策略）
```

### 常见标注错误

| 任务描述 | 易错标注 | 正确标注 | 原因 |
|---------|---------|---------|------|
| "设计缓存策略" | low | high | 需要架构思考，不是机械操作 |
| "修复登录 bug" | low | medium | 需要推理和测试，不是简单替换 |
| "重构支付系统" | high | xhigh | 跨多个模块，影响面大 |
| "格式化代码" | medium | low | 完全机械，无需思考 |
